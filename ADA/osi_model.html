1. How to Use These Notes

The lecture is organized around the 5-layer TCP/IP stack (application, transport, network, data-link, physical), while also referencing the broader OSI framing. The key to mastering this material is to keep asking one question: what problem is each layer trying to solve?

  • Application layer: What task does the user want to perform?
  • Transport layer: How do large conversations stay organized, reliable, and separated by app?
  • Network layer: How does data find the right destination across multiple networks?
  • Data-link layer: How do devices communicate reliably on the same local network?
  • Physical layer: How do raw bits become signals on real media?
Study strategy: First learn the purpose of each layer, then learn the specific mechanisms (for example: CRC at data-link, IP addressing at network, sequence numbers at transport, HTTP at application).
Key Takeaway:
Treat the course as a chain of problem-solving layers, not as a pile of disconnected protocols.

2. Core Networking Ideas

2.1 Why computer networking matters

The deck emphasizes that a common set of networking technologies supports many applications: email, instant messaging, browsing, and client-server systems. Because the underlying infrastructure is shared, a single laptop can perform all of these roles using the same network stack.

Another major lecture claim is that the basic principles of networking have remained stable for decades. Even though devices and speeds improve, the fundamental design logic - efficient sharing, layering, addressing, error control, and signal transmission - remains the same.

Real-world example: Your laptop can send an email, join a Zoom call, and load a website at the same time because those applications all reuse the same TCP/IP foundations.

2.2 The overarching design principle: efficient resource use

The lecture frames networking around resource utilization efficiency, especially efficient use of communication links ("the cable plant"). Link capacity is expensive, so it is worth adding technical complexity - such as routers and switching logic - if that complexity improves how well the network uses shared media.

A related secondary principle is to keep end-user equipment simple and hide complexity in dedicated network devices.

Real-world example: Home users do not manually manage packet routing or collision handling. Those tasks are pushed into routers, switches, and network interface hardware.

2.3 Packetization

Packetization means breaking user data into smaller pieces and packaging those pieces so they can travel across the network and later be reassembled. Each packet has:

  • Payload: the actual user data.
  • Overhead: extra control information needed for delivery and reconstruction.

The lecture explains that packetization is preferred because communication links are expensive relative to switching. This makes it better to share links efficiently, even if routers must do more work.

Real-world example: Shipping a whole warehouse as one giant container is risky and inflexible. Sending many labeled boxes is easier to route, recover, and redistribute. Networking works the same way.

2.4 Why packetization is useful

  • Improves communication-link efficiency.
  • Makes it practical to use dedicated routers.
  • Helps eliminate single points of failure.
  • Makes data easier to transport and re-route.

2.5 Layering

Layering organizes networking functions hierarchically: lower layers provide services that support higher layers. The lecture compares this to a corporate structure (workers support managers, managers support executives) and to computing (applications rely on operating systems, which rely on drivers).

The biggest advantage is modularity. One layer can evolve without forcing changes in all other layers. For example, the deck explicitly notes that moving from IPv4 to IPv6 does not require rewriting web browsers or email clients, and adding wireless networking does not change website URLs.

Real-world example: You can switch from Wi-Fi to Ethernet and still open the same website because application behavior stays stable even when lower-layer technology changes.

2.6 The five core communication tasks and the header idea

The lecture reduces networking to five recurring tasks. Packets solve these tasks by carrying headers.

Networking task What it means Header information used
Specify user needs Describe what the application wants to do User commands
Segmentation and reassembly Break large data into smaller pieces, then restore it Sequence numbers
Identifying and locating destination Figure out where the data should go Address
Error control Detect damage and sometimes trigger correction Error check
Signaling Convert data into a transmittable physical form None at this abstract level

The central design idea is that every layer adds the header information needed for its own task. Routers and end systems inspect these headers to decide how to process the packet.

2.8 Core Internet protocol principles

  • The Internet is a network of independent networks.
  • Each underlying network may use different local technology.
  • Many different applications should be able to reuse the same shared transport and network layers.
  • Routers benefit from a standardized message block because they can forward traffic in a uniform, "cookie-cutter" way regardless of application.

Real-world example: A company LAN, a university network, and a cell carrier can all be different internally, yet IP lets traffic move between them.

2.9 Standards and RFCs

The deck points students to RFCs (Requests for Comment) as the public standards source for Internet technologies. For exams, know that RFCs are where protocol behavior is formally specified.

Key Takeaway:
The entire networking stack exists to move data efficiently by splitting the problem into reusable layers, each of which adds just enough control information to do its own job.

3. Physical Layer

3.1 What the physical layer does

The physical layer provides transparent transmission of a bit stream over an actual communication medium. This is the layer that interacts directly with nature. It is where bits become electrical, optical, or radio signals.

The lecture stresses that the physical layer is special because we are "at the mercy of nature." The properties of the medium determine what signals are possible.

Real-world example: A bit sent through copper becomes a voltage pattern; the same bit sent through fiber becomes a light pattern.

3.2 Data vs. signal

Data is the information itself (facts, characters, numbers, ideas). Signals are the detectable transmitted energy used to carry that information.

  • Data in a spreadsheet is information.
  • The voltage on a wire carrying that spreadsheet's bits is a signal.

This distinction matters because the lecture explicitly says data cannot be transmitted directly; it must first be converted into signals.

3.3 Media: copper and fiber

The two main media emphasized are copper wire and optical fiber.

Copper / UTP

In common networking cable, eight copper strands are organized into four twisted pairs. This is UTP (Unshielded Twisted Pair). Twisting reduces interference caused by induction. Tighter twists support higher data rates.

Pair Color code Use in the slide
Pair 1White-Blue / BlueNot assigned in this example
Pair 2White-Orange / OrangeTransmit data
Pair 3White-Green / GreenReceive data
Pair 4White-Brown / BrownNot assigned in this example

The deck contrasts Cat 3 (slower) and Cat 5/Cat 5e (faster), and notes that Cat5e commonly uses RJ-45 connectors.

Optical fiber

Optical fiber is described as a thin strand of glass that guides light. It can carry large volumes of data across long distances, weighs less than copper, and is the dominant medium for long-distance traffic.

Real-world example: Inside a building, devices may connect over copper Ethernet. The building's aggregated traffic is then often carried over fiber to reach the broader network.

3.4 Noise and why binary signaling matters

Noise can distort signals. For communication to be reliable, the receiver must still decide what bit pattern was intended. The slides argue that binary signaling is more robust than multi-level signaling because fewer signal levels are easier to distinguish in the presence of noise.

The lecture compares a 2-level system (binary) against a 3-level system (ternary) and explicitly states that ternary systems are more error-prone.

Real-world example: It is easier to decide whether a light is "off" or "on" than to reliably distinguish among three close brightness levels in a noisy environment.

Diagram walkthrough: Binary vs. ternary detection
  • In the binary case, the receiver uses one decision threshold: below the line means 0, above means 1.
  • In the ternary case, the receiver needs multiple thresholds and narrower ranges.
  • As noise increases, those narrower ranges create more classification errors.

3.5 Shannon-Hartley theorem

A visual slide introduces Shannon's theorem to answer a central question: How much data can a medium carry?

Formula shown in the deck:
C = B * log2(1 + S/N)
where C is channel capacity in bits/second, B is bandwidth in Hertz, and S/N is the signal-to-noise ratio (SNR).

The following example slide applies the formula to a phone line:

  • B = 3400 Hz
  • SNR = 45 dB, so linear SNR is approximately 10^(45/10) = 31,622
  • Capacity is then about 51,000 bps (51 kbps)

The lecture explicitly states that this is the maximum data rate of phone modems.

Real-world example: Even if software is perfect, a phone line cannot exceed the physical information limit set by bandwidth and noise.

3.6 Bandwidth and Hertz

The deck defines Hertz as cycles per second. A standard phone line example is given as 0 Hz to 3400 Hz, which yields a bandwidth of 3400 Hz.

3.7 How signaling works end-to-end

The lecture's "Hello" example breaks signaling into a four-step path:

  1. Encode letters as binary numbers.
  2. Modulate a carrier signal using those binary numbers.
  3. Demodulate at the receiver to recover binary numbers.
  4. Decode the binary back into letters.

ASCII is presented as the standard encoding for common English characters (0-127), while Unicode is used for nearly all languages and far larger character sets.

Real-world example: Typing "Hello" into a browser becomes character codes, then bits, then signals, and only later becomes text again on the receiving machine.

Diagram walkthrough: Amplitude modulation signaling

The visual slide shows a stream of bits mapped onto a wave whose amplitude varies over time. The core point is not the exact waveform shape; it is that bit values are encoded into a physical signal pattern that moves through the medium.

3.8 Receiver operation

The receiver tracks bit periods. If the expected wave is present in the bit window, it interprets a 1; otherwise it interprets a 0. Those bits are then decoded back into characters.

3.9 Multiplexing

The section summary states that multiplexing enables efficient use of media. Even though the slide text shown to you is brief, the exam idea is simple: one physical medium can carry more than one logical stream when the signals are organized appropriately.

Real-world example: A single fiber backbone can carry aggregated traffic from many users at once instead of one dedicated line per person.

Key Takeaway:
The physical layer is about converting bits into signals that survive real-world limits such as noise, bandwidth, and the properties of copper or fiber.

4. Data-Link Layer

4.1 What the data-link layer does

The data-link layer transfers data between devices on the same network and detects (and sometimes helps correct) errors introduced below it. In this deck, the dominant example is Ethernet.

Real-world example: Two devices on the same office LAN communicate using Ethernet framing even before any Internet-wide routing is involved.

4.2 Ethernet basics

Ethernet is presented as the most common end-user data-link technology. The lecture describes it as:

  • Low-cost
  • High-speed
  • Designed for relatively small local networks

The deck gives one reference scale: an Ethernet network can be roughly 100 meters in radius with up to about 250 devices.

Diagram walkthrough: Early Ethernet

The historical diagram shows a shared "ether" cable with multiple stations attached to a common medium. This helps explain why early Ethernet naturally behaved as a shared broadcast medium: every attached station could "see" the traffic on the same cable.

4.3 Ethernet transmission and broadcast

When one machine (say PC A) wants to send to another (PC B), the sender first adds the destination's Ethernet address. The lecture compares this to writing a "To:" address on an envelope.

But unlike postal mail, classic Ethernet uses broadcast behavior: the signal is placed on the shared medium, so all attached stations receive the frame. Only the device whose address matches the destination should accept it at the higher layers.

This design makes delivery simple, but every machine must spend effort inspecting frames that are not intended for it. That is why the lecture notes Ethernet cannot scale indefinitely as one giant shared broadcast domain.

Real-world example: In a shared classroom LAN, every device can physically hear the transmission, but only the addressed NIC should treat the frame as relevant.

4.4 CSMA/CD and collisions

If multiple senders talk at the same time on shared Ethernet, their signals interfere. This is a collision. The lecture's solution is CSMA/CD: Carrier Sense Multiple Access with Collision Detection.

  • Carrier sense: transmit only when the medium seems quiet.
  • Multiple access: many devices share the same medium.
  • Collision detection: keep listening while sending; if a collision is detected, stop.
  • After a collision, wait and retry later.

Real-world example: It is like two people on a walkie-talkie channel: listen first, speak only if quiet, and if both speak at once, stop and try again.

Diagram walkthrough: Collision on shared Ethernet

The visual shows two PCs injecting data onto the same Ethernet backbone at the same time. Their frames overlap in the medium, producing a collision. The point of the picture is to show that the conflict happens on the shared cable itself, not inside either endpoint.

4.5 Error detection and why naive checks fail

Frames can be damaged by physical imperfections such as noise and power disturbances. The lecture first shows a simple "sum the digits" check and then explains why it is too naive: different corruptions can produce the same check value (error cancellation).

This motivates stronger error detection.

4.6 CRC (Cyclic Redundancy Check)

Ethernet uses CRC-32 for frame error detection. The lecture emphasizes that CRC is the practical, real-world standard used by commercial data communication technologies.

  • CRC-32 is described as 99.99999998% reliable for errors longer than 32 bits.
  • It detects all errors affecting fewer than 33 bits.
  • It also detects all odd-number-of-bit errors.

CRC sender logic

  1. Treat the user data as the dividend; the technology specifies a divisor of n + 1 bits.
  2. Append n zeros to the data.
  3. Perform modulo-2 division using XOR-style subtraction.
  4. Append the remainder as the CRC field and send the frame.

CRC receiver logic

  1. Divide the received data (including CRC) by the same divisor.
  2. If the remainder is 0, accept the data; otherwise reject it.
Diagram walkthrough: CRC sender operation

The diagram labels four essential pieces: user data, the zeros appended, the divisor, and the resulting CRC remainder. Read it left to right as: append zeros -> divide -> keep the remainder -> attach the remainder to the outgoing frame.

4.7 Ethernet frame structure

By the time the lecture reaches frame structure, you have already seen destination address and CRC. The frame includes other fields so the receiver knows when a frame starts, where it came from, and where it ends.

The preamble pattern shown in the deck is repeating 10101010, and the start frame delimiter is 10101011.

Diagram walkthrough: Ethernet frame layout
  • The frame begins with synchronization fields (preamble + SFD).
  • Address fields tell the NIC who sent the frame and who should receive it.
  • The payload is the data coming down from upper layers (for example, an IP packet).
  • The frame ends with the FCS, which lets the receiver test whether the frame was damaged in transit.
  • The slide also notes that bits within the frame are transmitted least significant bit first.

4.8 Ethernet addressing

Ethernet addresses are 48 bits long. The lecture notes:

  • All 1s in the destination address is the predefined broadcast address on the LAN.
  • Addresses can be universally administered (assigned by manufacturer).
  • They can also be locally administered (assigned by a network administrator).

The lecture also shows that MAC addresses are commonly written in hexadecimal: 48 bits are broken into twelve 4-bit groups, each represented by a hex digit.

Real-world example: A factory-assigned NIC address is like a serial number burned into the hardware.

4.9 Hubs vs. switches

The deck explicitly contrasts old hubs and modern switches:

  • Hub: sends data out to all connected computers.
  • Switch: sends data only toward the intended destination.

This makes switching faster and much more efficient. The switch learns which devices are on which ports by reading the source addresses of incoming frames and building a forwarding table.

Real-world example: A hub is like shouting the same message to every room in a building. A switch is like handing it only to the correct office.

Key Takeaway:
The data-link layer makes local communication practical by packaging data into frames, handling collisions on shared media, and detecting corruption with CRC.

5. Network Layer

5.1 What the network layer does

The network layer moves variable-length packets from a source network to a destination network through one or more networks. The core job is routing, and the devices that perform it are routers.

Real-world example: Sending a packet from your home Wi-Fi to a university server involves multiple separate networks, so local Ethernet is not enough. Routing is required.

5.2 Routing as a "road trip"

The lecture compares routing to a long road trip:

  • The source and final destination are like network-layer addresses.
  • Intermediate exits and local turns resemble lower-layer, local addressing decisions.

This is a strong exam analogy because it separates the big-picture destination from the local hop-by-hop mechanics.

5.3 Internet Protocol (IP)

The deck presents IP as the dominant network-layer protocol. It is specified in RFC 791 and is now used by default in all operating systems.

Conceptually, IP is responsible for adding source and destination addresses so routers can move data toward the right network.

5.4 IP packets live inside lower-layer frames

A key layering idea is that an IP packet is carried inside a data-link frame. In other words, the network layer relies on the data-link layer to move each hop locally.

Real-world example: An IP packet crossing the Internet may be wrapped in one Ethernet frame on your LAN, then another frame type somewhere else, while the IP destination stays the same.

5.6 IP addressing fundamentals

An IP address is a unique label used to locate a network interface. In IPv4, the source and destination address fields are 32 bits each.

The lecture explicitly distinguishes IP addresses from MAC addresses:

  • MAC addresses are hardware-oriented link-layer identifiers.
  • IP addresses are assigned by administrators and are flexible.

5.7 Binary numbers and dotted decimal notation

Because IP is fundamentally binary, the lecture spends time on decimal-to-binary and binary-to-decimal conversion. The exam implication is clear: you should be comfortable with 8-bit binary values.

IPv4 addresses are written in dotted decimal notation: 32 bits are split into four 8-bit chunks, each chunk is converted to decimal, and the four decimals are separated by dots.

Example from the deck style: 192.168.1.5

5.8 IP address structure: network part + host part

A major conceptual point in the lecture is that IP addresses are structured, not assigned randomly. They are split into:

  • Network part: identifies the organization or network.
  • Host part: identifies the individual device within that network.

The lecture reinforces this using analogies:

  • Street address: street name = network; house number = host.
  • Phone number: area code / exchange / local number reflect hierarchy.
  • Credit card number: issuer information first, account-specific bits later.
  • ZIP code: broader region first, finer destination later.

Real-world example: A university may own a network block, then allocate individual host addresses inside that block to labs, offices, or departments.

5.10 How organizations obtain IP addresses

The lecture lists the major regional registries that distribute IP space: ARIN, RIPE NCC, APNIC, LACNIC, and AfriNIC.

Large carriers generally receive large blocks directly because that simplifies routing. Most organizations then receive addresses from those carriers (ISPs), rather than directly from the top-level registries.

5.11 IPv4 exhaustion and why IPv6 exists

The lecture states that IPv4's main limitation is that the address pool is running out. IPv4 provides 2^32 = 4,294,967,296 addresses - about 4 billion total.

The deck also notes that allocation inefficiency means the world can run out of usable IPv4 space well before 100% of the raw pool is productively used.

5.12 IPv6 overview

IPv6 is introduced primarily as the answer to address exhaustion. It is defined in RFC 2460 and:

  • Expands source and destination addresses
  • Simplifies router processing
  • Removes the IPv4 header checksum
  • Adds features such as flow labels

The new address pool is 2^128, which the lecture emphasizes by noting the enormous number of possible addresses per square meter of the Earth's surface.

5.14 IPv6 notation and compaction rules

IPv6 addresses are written as 8 groups of 4 hexadecimal characters. Each hex digit represents 4 bits, which is why eight groups of four digits can represent 128 bits total.

The deck's compaction rules are important exam material:

  • Omit leading zeros inside a 16-bit field.
  • A field of 0000 should be written as 0.
  • :: represents consecutive zero fields and can appear only once.
  • It must be used to compress the largest possible run of zero fields.
  • If there is a tie, use the first possible occurrence.
  • Hex letters a-f should be lowercase.

Example from the deck: 2001:0db8:aaaa:0000:0000:0000:eeee:0001 becomes 2001:db8:aaaa::eeee:1.

Key Takeaway:
The network layer is the addressing and routing layer: it gives packets a global destination and guides them across multiple networks, with IP as the core mechanism.

6. Transport Layer

6.1 Why the transport layer is needed

The lecture's starting point is that IP is only best effort. It can send packets toward the correct destination, but packets may be lost, duplicated, damaged, or arrive out of order.

This means most applications cannot rely on IP alone. The transport layer adds the missing controls: segmentation, reliability, application separation, and flow regulation.

Real-world example: Downloading a 5,000,000-byte file over IP alone would be unsafe and awkward. Transport protocols break it up, track it, and make sure the receiver reconstructs it correctly.

6.2 The transport layer's place in the stack

The transport layer sits between applications and IP. Many applications (HTTP, SMTP, SIP, and others) can feed into the same transport layer, which then passes their traffic into IP.

This is one of the clearest examples of layering: many application protocols share the same transport service.

Diagram walkthrough: Transport layer in the stack

The diagram shows multiple applications converging onto one transport layer, then down into IP, then down into the local network. It visually explains why the transport layer is the shared "middle manager" for end-to-end communication.

6.3 TCP overview

TCP (Transmission Control Protocol) is the primary transport protocol emphasized in the lecture. It is defined in RFC 793 and described as a highly reliable host-to-host transport protocol over packet-switched networks.

The send/receive workflow is:

  1. The application gives data to TCP.
  2. TCP breaks it into segments and adds sequence numbers.
  3. IP carries those segments onward.
  4. The receiver's TCP collects segments, reorders them, and reassembles the original data.

6.4 TCP's five major jobs in this deck

  • Segmentation
  • Reliability
  • Multiplexing
  • Flow control
  • Connection establishment (named in the overview and summary, though not expanded into a dedicated slide here)

6.5 Segmentation and sequence numbers

TCP allows applications to send data larger than one IP packet by breaking the data into segments. Each segment receives a sequence number.

Sequence numbers help the receiver:

  • Reassemble segments in the correct order
  • Detect missing data
  • Detect duplicate segments

The lecture gives a concrete rule: next sequence number = previous sequence number + previous segment length.

Real-world example: It is like numbering pages in a printed packet so the receiver can restore the correct order even if pages arrive mixed up.

6.6 Reliability: ACKs, timeouts, and retransmission

TCP recovers from loss or damage using acknowledgments:

  • If data arrives correctly, the receiver sends a positive ACK.
  • If the sender does not receive that ACK within a timeout, it retransmits.
  • If duplicates show up, sequence numbers make them recognizable.

The deck also notes that TCP uses a checksum to detect serious damage. Damaged segments are discarded, and the sender eventually retransmits because the expected ACK never arrives.

Real-world example: It is like sending certified mail and resending if the delivery confirmation never returns.

6.7 Multiplexing and ports

Modern computers run many networked activities at once, so TCP uses port addresses to separate them. Each port acts like a separate communication channel within one host.

An IP address plus a port is a socket. A pair of sockets uniquely identifies a connection.

Diagram walkthrough: TCP ports and airport gates

The slide compares a computer to an airport. The computer's IP address is like the airport code, while each port is like a gate. Multiple flights (connections) can pass through the same airport at the same time because each one is assigned its own gate.

The OS chooses an available sender-side port, but the receiver must listen on a known port if clients are to find it.

Standard ports from the lecture

It also notes that port assignments are standardized by IANA, and that service lists can be found in /etc/services on Unix/Linux or the corresponding Windows services file.

Since port numbers are 16 bits long, the lecture points out that there are 65,536 possible ports per host.

6.8 Flow control and the sliding window idea

Flow control exists because a slow receiver can become overwhelmed if the sender transmits too quickly. TCP solves this by letting the receiver regulate how much data may be sent before more acknowledgment is required.

The simplest idea is stop-and-wait: send one segment, wait for ACK, send the next. The lecture says this works but is very slow.

The more efficient design is to return a window with each ACK. The window tells the sender how many bytes may be sent before waiting for more permission.

Diagram walkthrough: TCP flow control with window size
  • The receiver advertises a window (example shown: 700 bytes).
  • The sender sends multiple segments as long as the total bytes stay within that window.
  • In the pictured example, segment 1 has sequence 1 and length 300; segment 2 has sequence 301 and length 320.
  • The receiver acknowledges up through byte 620 with ACK 621, meaning the next expected byte is 621.

This creates the sliding window: the permitted send range moves forward as the receiver acknowledges data or changes the allowed window size.

Real-world example: Think of a conveyor belt where the receiver controls how many boxes may be placed on the belt before asking the sender to pause.

6.10 UDP: when less is more

UDP is defined in RFC 768. The lecture frames it as the lighter-weight option for applications that do not need TCP's full overhead.

By avoiding sequence tracking, window management, and related control state, UDP reduces protocol overhead.

Real-world example: A simple query/response or latency-sensitive exchange may prefer lower overhead over strict recovery behavior.

Important exam nuance: The lecture summary mentions "why three-way handshake", but the uploaded deck portion does not include a dedicated handshake explanation slide. You should recognize it as part of TCP connection establishment, but the detailed handshake sequence is not explicitly spelled out in the provided content.
Key Takeaway:
The transport layer turns best-effort packet delivery into usable application communication by handling order, reliability, port separation, and receiver-driven flow control.

7. Application Layer

7.1 What the application layer does

The application layer defines the end-user activity and the commands used to support that activity. This is the layer the user directly interacts with.

The lecture explicitly notes that end users only "see" application interfaces. They do not directly see TCP or IP.

Real-world example: When a user clicks a link, they think "open this page," not "send TCP segments and route IP packets."

7.2 Application protocols vs. lower-layer protocols

Unlike lower layers, where a smaller number of protocols provide broadly reusable services, the application layer contains different protocols for different end-user tasks.

  • SMTP for email transfer
  • HTTP for the web
  • FTP for file transfer
  • Telnet for remote login (the lecture notes it is now superseded by SSH)

All of these still rely on the same underlying TCP or UDP services.

7.3 HTTP and the Web

The lecture says the web is the "lay person's interpretation of the Internet." In practice, that means the most visible Internet activity for everyday users is retrieving information using HTTP.

The deck's examples include shopping, social sites, and assignments, which all reinforce the same idea: HTTP is a general-purpose content retrieval protocol used everywhere.

7.4 Web structure and hyperlinks

The web is described as a set of pages linked by hyperlinks. Because linked pages can be anywhere in the world, the collection forms a "world wide web."

The key functional feature of a web page is that one information item can immediately lead to another.

Real-world example: Clicking "Those who bought this also bought..." follows hyperlinks that connect related pages and data.

7.5 HTML as the page format

Web pages are written in HTML (HyperText Markup Language). The example in the deck shows the basic structure:

  • <html> wraps the document
  • <head> contains metadata like the title
  • <body> contains visible content
  • <a href="..."> creates a hyperlink

Real-world example: A heading that links to a university homepage uses an anchor tag so a click triggers a new HTTP request.

7.6 HTTP as request/response

The lecture defines HTTP (via RFC 2616 in the slide set) as a general-purpose data transfer protocol based on a request/response model.

  • The client sends a request.
  • The server returns a status code, meta-information, and the data.
Diagram walkthrough: HTTP example

The visual request/response transcript can be read in three blocks:

  1. Request line + request headers: for example GET /index.html HTTP/1.1, host, accepted formats, encoding preferences, cookies, and connection preferences.
  2. Status line + response headers: for example HTTP/1.1 200 OK, date, server, content length, content type, and related metadata.
  3. Body: the returned HTML document itself.

This picture matters because it makes clear that HTTP transfers not just content, but also a large amount of metadata about the content.

7.7 HTTP features the lecture wants you to notice

  • HTTP responses include meta-information about the transferred data.
  • The client can know the content type in advance (text, PDF, JPG, etc.).
  • The transfer can be modified in transit at the protocol level, for example through compression.
Diagram walkthrough: Content-Encoding for compression

The compression slide highlights the response header Content-Encoding: gzip. The important point is that the server can tell the client the body has been compressed, so the client knows it must decompress before interpreting the content.

7.8 URLs (Uniform Resource Locators)

A URL is the address used to access a web resource. The deck cites RFC 1738 and gives the syntax:

protocol://host[:port]/[abs_path[?query]]

This breaks a web address into:

  • Protocol: how to communicate (for example, HTTP)
  • Host: which server to contact
  • Port: which service endpoint on that server (optional if default)
  • Path: which resource on the server
  • Query: extra request parameters
Example What it means
http://www.usf.edu:8080/acad_calendar.html Fetch a specific page from host www.usf.edu on port 8080
http://www.usf.edu:8080/grades?fName=joe Fetch from a path while supplying a query parameter
http://www.google.com/search?q=usf Search query with q=usf

7.9 URL defaults

Web servers often recognize default ports and default files. The lecture gives the example:

http://www.usf.edu is treated the same as http://www.usf.edu:80/index.asp

It also notes other default filenames such as index.html and index.php.

The lecture finally shows that URL-style syntax can describe non-web network resources too, such as a database connection.

Key Takeaway:
The application layer is where user-facing tasks live. HTTP, HTML, hyperlinks, and URLs are the web-specific pieces that make the Internet usable to ordinary users.

9. Exam Review Sheet

9.1 One-line layer memory aid

Physical: turn bits into signals.
Data-link: move frames on one local network and catch damage.
Network: address and route packets across networks.
Transport: make delivery usable for applications.
Application: define the user-visible service.

9.2 High-yield facts to memorize

  • Packetization = payload + overhead.
  • Layering = lower layers support higher layers.
  • Binary signaling is more robust than ternary in noise.
  • Shannon-Hartley gives the theoretical capacity limit of a channel.
  • UTP uses four twisted copper pairs; Cat5e commonly terminates in RJ-45.
  • Ethernet is local, historically shared, and originally broadcast-based.
  • CSMA/CD = listen first, detect collisions, stop, wait, retry.
  • CRC is the practical Ethernet error-detection method.
  • MAC is a local link-layer address; IP is the routable network-layer address.
  • IPv4 is 32-bit; IPv6 is 128-bit.
  • TCP adds sequence numbers, ACKs, ports, windows, and more.
  • UDP keeps only basic fields and avoids most TCP overhead.
  • HTTP is request/response and carries both data and metadata.
  • URL syntax = protocol + host + optional port + path + optional query.

9.3 Likely conceptual questions

  • Why is packetization better than sending one giant block?
  • Why does layering make upgrades easier?
  • Why does binary signaling resist noise better?
  • Why can Ethernet broadcast on a LAN but not across the whole Internet?
  • Why is CRC better than a naive checksum idea?
  • Why do IP addresses have a network part and a host part?
  • Why is TCP needed if IP already exists?
  • Why do we need ports if each computer already has an IP address?
  • Why is sliding window better than stop-and-wait?
  • Why is HTTP more than "just the page body"?

9.4 The shortest possible full-stack story

A user clicks a link. The application layer creates an HTTP request. The transport layer (usually TCP) splits it into segments, numbers them, and manages reliability. The network layer gives those segments IP addresses so routers can move them across networks. The data-link layer wraps each local hop in frames with MAC addresses and CRC. The physical layer turns the bits into electrical, optical, or radio signals. The receiver reverses the whole process.

Key Takeaway:
If you can explain that one full-stack story clearly, you understand the architecture of the lecture.
