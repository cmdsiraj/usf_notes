1. Course Overview

This slide deck is really two connected mini-units:

  • Part A: Network security - how organizations protect data as it enters and leaves networked systems.
  • Part B: Computing infrastructures - the hardware, operating-system, and platform concepts that make modern digital services possible.

The bridge between the two parts is simple: security depends on infrastructure. You cannot secure systems well unless you understand what is being secured - applications, operating systems, networks, virtual machines, cloud platforms, and the data that moves among them.

How the material is organized conceptually:
1) Define what must be protected, 2) identify what can go wrong, 3) apply controls, and 4) scale those controls across real computing environments.
Key Takeaway
The deck is not just about “hacking” or “cloud.” It teaches a full chain: information security goals -> network threats -> controls -> infrastructure needed to run secure systems at scale.

2. Information Security Foundations

2.1 Network security sits inside information security

The deck defines network security as one part of the broader field of information security. Information security asks a basic question: what properties must information have to remain useful and trustworthy? The answer is the classic CIA triad:

Security GoalMeaningWhat failure looks likeReal-world example
Confidentiality Only authorized people or systems should be able to view the data. A private file, password, or customer record is exposed to unauthorized parties. A payroll spreadsheet is leaked because it was emailed without encryption.
Integrity Data should not be improperly changed, destroyed, or forged. A value is altered, a message is tampered with, or the source cannot be trusted. A bank transfer amount is changed while data is moving between systems.
Availability Authorized users should be able to access systems and information when needed. A system is down, blocked, overloaded, or unusable. A hospital system becomes unreachable during a ransomware incident.

2.2 Why information security matters

The deck emphasizes that modern economies and organizations are increasingly dependent on digital information processing. Many business records are stored only in computer systems, and many workflows cannot continue when those systems fail. That means information security is no longer a technical “nice to have”; it is a business continuity requirement.

Example: If a company keeps purchase orders, inventory counts, and customer invoices only in software, then a loss of availability does not just inconvenience the IT team. It can stop shipping, billing, customer support, and cash flow.

2.3 Vulnerabilities, threats, and controls

The deck’s general security model separates three core ideas:

  • Vulnerability: a weakness that can be exploited.
  • Threat: the attacker’s capability, intent, and method.
  • Control: the safeguard that blocks, reduces, or detects the attack.

Diagram walkthrough: General information security model

Figure ElementWhat it representsHow to interpret it
Central IT system / information assets The data and systems the organization cares about. This is the “thing of value” being protected.
Vulnerability inside the system A weakness such as insecure software, poor configuration, or weak access control. A threat becomes dangerous only when a vulnerability makes exploitation possible.
Threat arrows pointing toward the system Possible attacks coming from adversaries. Threats can come from many directions and use different methods.
Security controls shown as a protective barrier Preventive or mitigating safeguards. Controls reduce the chance that a threat reaches the vulnerable asset successfully.
“Blocked threat” vs. “successful attack” Two possible outcomes of the same attack attempt. If controls are strong enough, the attack is stopped; if not, the threat exploits the vulnerability.
Important logic: A threat by itself does not automatically cause damage. Damage occurs when a threat successfully exploits a vulnerability and no adequate control prevents it.
Key Takeaway
The foundational model is: protect information by preserving confidentiality, integrity, and availability, while reducing vulnerabilities, anticipating threats, and applying controls.

3. Network Security Controls

3.1 What network security adds

The deck defines network security as providing information security in the presence of risks created by computer networks. Networks matter because they expose systems to two directions of risk:

  • Incoming risk: attackers can send traffic into an organization to read, alter, or disable systems.
  • Outgoing risk: data leaving the organization can be read, modified, or blocked in transit.
Example: An external attacker probing an employee portal is an incoming threat. A customer’s payment data crossing the internet without protection is an outgoing risk.

3.2 Why network security matters

The slides stress that critical infrastructure - such as power systems and payment systems - is network-connected. That means weak network security can create very expensive failures. In other words, the network is not just a convenience layer; it is an attack surface.

3.3 Control matrix by security objective

One of the most important summary slides classifies controls by whether they protect incoming or outgoing information and by which part of the CIA triad they support.

Security GoalIncoming Information ControlsWhy they helpOutgoing Information ControlsWhy they help
Confidentiality Patching, authentication, authorization These reduce unauthorized entry and limit who can access data once inside. Encryption Encryption protects data from being read while it travels over untrusted networks.
Integrity Firewalls Firewalls regulate traffic so malicious or unauthorized packets are blocked. Digital signatures Digital signatures allow the receiver to detect tampering and verify origin.
Availability Virus protection, end-user training These reduce malware infections and risky user behavior that can disrupt services. Redundancy Backup paths and spare capacity keep services running despite failures.

3.4 Patching

Patching means applying software updates that fix known vulnerabilities. Since software is complex, vulnerabilities are common, and known flaws often become public knowledge. Once a flaw is known, attackers frequently automate attempts to exploit systems that have not been updated.

Example: A web server with an unpatched remote-code-execution bug may be compromised within hours of the bug becoming widely known, especially if attackers are scanning the internet for that version.

Small organizations may rely on automatic updates, but the key idea is speed: the longer a patch is delayed, the longer the vulnerability remains exploitable.

3.5 Authentication and authorization

Authentication asks, “Who are you?” Authorization asks, “What are you allowed to do?” These are related but different.

  • Authentication verifies a claimed identity, commonly with a username and password.
  • Authorization grants or denies permissions such as read, write, delete, or execute.

The slide uses a physical analogy: employees are given keys only to the rooms they need. The computing equivalent is the principle of least privilege: users should only get the minimum access needed for their job.

Example: A marketing employee may authenticate successfully to the company system, but authorization rules should still prevent them from editing payroll data.

3.6 Good passwords

The deck highlights several password rules:

  • Use more than just letters.
  • Avoid obvious names and dictionary words.
  • Use passwords longer than 8 characters.
  • Change them regularly.
  • Do not reuse them, especially across sensitive accounts.

The deeper lesson is that passwords fail not only because they are weak, but also because people reuse them. A breach on one site can become a breach on many sites if the same password is used everywhere.

Example: If a user reuses one password on a social site and a banking site, a single password leak can become a financial loss.

3.7 Firewalls

A firewall is a system placed between networks to regulate traffic. It examines packets entering or leaving a network and applies administrator-defined rules to allow or block them. Common rule criteria include:

  • Source IP address
  • Destination IP address
  • Destination port

The slides describe a simple rule-building process:

  1. Block insecure services (such as telnet or FTP).
  2. Block blacklisted networks.
  3. Allow trusted services (such as SSH).
  4. Allow safe public services (such as web, DNS, and email).

Diagram walkthrough: Common firewall / DMZ design

ZoneWhat lives thereWhy it is designed this way
Internet External users and untrusted networks. This is the open environment from which attacks may originate.
Outer firewall Boundary between the internet and public-facing services. It blocks clearly unsafe traffic before it reaches exposed servers.
DMZ (de-militarized zone) Public-facing services such as web, DNS, and email. These services must be reachable from outside, but they should be separated from the internal network.
Inner firewall Boundary between the DMZ and the internal network. If a public server is compromised, the attacker still faces another barrier before reaching internal systems.
Internal network Private organizational systems and sensitive data. This is the most protected zone and is blocked from direct outside access.
Example: A company website may be reachable by everyone on the internet, but its internal HR database should never be directly reachable from the public web.

3.8 Anti-virus and end-user training

The deck treats security as both a technical and a human problem.

  • Anti-virus / anti-malware tools detect or block malicious programs such as viruses and worms.
  • End-user training teaches people to avoid dangerous behavior, especially with suspicious email, macros, and phishing pages.

The slide distinguishes worms (self-contained and self-spreading) from viruses (which use other programs to replicate). The practical lesson is that malware can spread quickly, so signatures and detection tools must be kept current.

Example: An employee who opens a malicious spreadsheet attachment can trigger malware even if the core network controls are strong. User behavior is part of the attack surface.
Key Takeaway
Network security controls are easiest to remember when grouped by purpose: prevent entry, limit access, regulate traffic, protect data in transit, and reduce disruption.

4. Encryption, SSH, and Integrity Protection

4.1 What encryption is

Encryption converts readable plaintext into unreadable ciphertext so that only someone with the right key can recover the original information. The deck emphasizes two building blocks:

  • Algorithm - the procedure used to transform data.
  • Key - the secret (or partially public) value that controls the algorithm’s behavior.

Diagram walkthrough: plaintext to ciphertext

StageSender viewWhat an eavesdropper seesReceiver view
Before encryption Message is readable (“Hello”). If intercepted now, the content is exposed. Not received yet.
After encryption Sender turns plaintext into ciphertext (“IFMMP” in the slide’s simple example). The attacker sees scrambled data that should not reveal the original message directly. Receiver later applies the correct reverse process.
After decryption Done sending. Still cannot interpret the protected transmission easily. Receiver recovers the original plaintext.

4.2 Symmetric key encryption

In symmetric encryption, the same key is used to encrypt and decrypt. This makes symmetric systems fast and computationally efficient. The slide gives a toy example: shifting letters by one position.

The current real-world standard mentioned is AES (Advanced Encryption Standard).

Main advantage: Symmetric encryption is fast, so it is good for protecting large amounts of data.
Main challenge: Both parties need the same secret key, so key exchange becomes a problem.
Example: Once a secure session starts, the bulk of an HTTPS connection is typically protected with fast symmetric encryption because sending every byte with a slower public-key operation would be inefficient.

4.3 Asymmetric key encryption

Asymmetric encryption uses two related keys:

  • Public key - shared openly and used for encryption (or verification).
  • Private key - kept secret and used for decryption (or signing).

This solves the “unsafe key exchange” problem of symmetric encryption because the public key can be sent openly.

The deck notes that many asymmetric systems use modular arithmetic. In its simplified numeric example, ciphertext is produced with:

ciphertext = plaintext * 3 mod 10

and decrypted with:

plaintext = ciphertext * 7 mod 10

Study table: simplified asymmetric-key mapping from the slide

Plaintext digitEncrypted digit (x * 3 mod 10)Meaning
000 maps to 0 in this toy system.
131 becomes 3.
262 becomes 6.
393 becomes 9.
42The modulo operation “wraps around.”
55Some values can map to themselves in toy examples.
68Pattern continues under modulo 10.
717 becomes 1.
848 becomes 4.
979 becomes 7.

In real systems, the numbers are far larger, which is what makes recovering the private key from the public key computationally difficult. The deck names RSA as a popular example.

4.4 Why both symmetric and asymmetric encryption are used together

The deck explicitly states that practical systems usually combine the two approaches:

  • Asymmetric encryption is used first to exchange a temporary symmetric session key.
  • Symmetric encryption is then used for the rest of the session because it is faster.

This gives the best of both worlds: safer key establishment plus efficient bulk encryption.

Example: When you open a secure banking site, public-key techniques help set up the session safely, and then the session continues with faster symmetric encryption.

4.5 TLS, VPN, and SSH

The deck gives three important applied uses of encryption:

TechnologyMain purposeHow to think about itTypical example
TLS Protect application traffic in transit. Application-level secure channel, commonly seen as https. Secure browsing, online banking, secure email transport.
VPN Create a protected path into an organization’s network. Traffic is routed through a trusted gateway inside the organization. Remote employee accessing internal files or systems from home.
SSH Secure remote login and secure command/data transport. An encrypted replacement for insecure remote terminal access such as telnet. Administrator securely logging into a Linux server.

Diagram walkthrough: TLS vs VPN

PatternWhat the figure showsInterpretation
TLS A user’s browser connects securely to a specific site (the slide contrasts a generic university site with a logged-in bank site). TLS protects a particular application session, not necessarily all device traffic.
VPN A remote laptop routes traffic through a VPN gateway before reaching internal organizational resources. VPN protects the path into the organization and extends access to internal resources that may not be publicly exposed.

4.6 SSH authentication and transaction states

The SSH slides emphasize two ideas:

  • Host authentication: clients need confidence they are talking to the correct server, not an impostor.
  • Negotiation before secure communication: SSH first uses public-key methods to negotiate algorithms and shared secrets, then switches to symmetric encryption for the actual session.

Study table: SSH transaction states

StageWhat happensWhy it matters
1. TCP connection The client first establishes a normal TCP connection. SSH runs on top of TCP, so transport must exist before security negotiation begins.
2. Protocol identification Client and server exchange protocol version and software details. Both sides confirm they are speaking the same protocol.
3. Key exchange Encryption algorithms and keys are negotiated using public-key techniques. This creates a shared secret without sending that secret openly.
4. SSH_MSG_NEWKEYS The protocol signals that both sides should begin using the newly negotiated keys. This marks the transition from setup to protected communication.
5. Encrypted communication Traffic now uses the negotiated shared-key encryption. Commands, data, and credentials are protected on the network.
Example: When a developer uses SSH to log into a cloud server, the first connection may prompt them to trust the server’s host identity. On later connections, SSH checks that the server still matches that known identity.

4.7 Digital signatures and message integrity

Encryption is mainly about confidentiality, but security also requires integrity. The deck introduces digital signatures for this purpose.

The simplified idea is:

  1. Compute a message digest (a fixed-size summary of the message).
  2. Encrypt the digest with the sender’s private key.
  3. The receiver uses the sender’s public key to verify the digest.
  4. The receiver also computes a fresh local digest of the received message and compares the two.

If the digests do not match, the message was changed or corrupted.

Diagram walkthrough: confidentiality + integrity using asymmetric keys

Security goalSender actionReceiver actionWhy it works
Confidentiality Encrypt the message with Bob’s public key. Bob decrypts with Bob’s private key. Only Bob should possess the private key needed to read the message.
Integrity / authenticity Compute a digest and encrypt that digest with Alice’s private key. Use Alice’s public key to verify the digest and compare it with a locally computed digest. If verification fails or the digests differ, the message was altered or did not come from the expected sender.
Example: Signed software updates use this same logic. Your device checks the publisher’s signature before trusting the update, which helps prevent tampered or fake installers.
Key Takeaway
Remember the clean separation: encryption protects confidentiality, digital signatures protect integrity and authenticity, and real systems usually combine symmetric and asymmetric methods.

5. Availability and Redundancy

5.1 What redundancy means

Redundancy means keeping extra capacity, alternate paths, or duplicate components so that service continues even when something fails. In the slide deck, redundancy is explicitly tied to availability.

This is especially common for network-facing services such as:

  • DNS (name resolution)
  • Web services
  • Email services

5.2 How to read the redundancy figure

The figure shown on the slide is a network graph with one highly connected central node and many linked surrounding nodes. Even though the diagram is not explained numerically, the intended lesson is conceptual:

Observed feature in the figureWhat it implies
A central node connected to many others Some network points can serve as major hubs or aggregation points.
Many outward links Traffic or service can be distributed across multiple paths or peers.
Clusters branching away from the center Large systems are often built from multiple interlinked components rather than one single machine.
Multiple surrounding nodes rather than one endpoint Redundancy often means “more than one place to go” if a component fails.
Example: If a company hosts its website on two servers behind a load balancer, one server can fail and users may still reach the site through the other server.
Exam tip: Redundancy is not mainly about secrecy or correctness. Its primary purpose is keeping systems available despite failure, overload, or maintenance.
Key Takeaway
For availability, the big idea is simple: do not depend on a single component when downtime is costly.

6. Modern Computing Foundations

6.1 The five classic components of a computer

The second half of the deck begins with the Von Neumann model, which describes a computer as a system built from five broad parts:

  • Input
  • Output
  • Memory
  • Central control part
  • Central arithmetic part

Diagram walkthrough: Von Neumann architecture

ComponentRole in the figureWhy it matters
Input device Feeds data or instructions into the system. The computer needs a way to receive commands and data from users or other systems.
Output device Presents results produced by the system. Computation is only useful if the result can be communicated back out.
Control unit Directs which instruction is executed and in what order. This is the coordination center of computation.
Arithmetic / Logic Unit (ALU) Performs calculations and logical comparisons. This is where core computational work happens.
Memory unit Stores data and instructions used by the CPU. Programs and intermediate values must be stored somewhere accessible.

The most important conceptual point is that modern programming languages still map back to this architecture. Variables, assignments, loops, and function calls may look high-level, but at a lower level they still become operations on memory, control flow, and arithmetic/logic.

Example: A simple statement like total = price * quantity becomes a sequence of low-level steps: load values from memory, perform multiplication in the ALU, and store the result back in memory.

6.2 Operating systems

An operating system (OS) performs common services needed by applications. Instead of every application directly controlling disks, memory, displays, and devices, the OS provides standardized functionality.

The slides specifically mention that operating systems implement system calls, which are the functions through which applications request OS services.

Common services the OS handles

  • Reading and writing data from storage
  • Printing or sending output to devices
  • Managing computer usage and auditing
  • Allocating memory and processor time
  • Coordinating access to hardware devices

Diagram walkthrough: OS as the middle layer

Layer in the figureMeaningStudy interpretation
Users People interacting with applications. Users do not directly control the hardware.
End-user application programs Apps that users run for real tasks. Applications depend on the OS for shared services.
Operating system Manages resources and provides services to applications. The OS is the control layer that mediates between software and hardware.
Hardware + controllers + memory CPU, storage, USB, display, and the physical resource layer. The OS coordinates access so multiple applications can use shared resources safely.
Example: When a document editor saves a file, it does not usually “speak disk hardware” directly. It calls the OS, and the OS handles the actual file-system and device operations.

6.3 Big data and the 5V model

The deck uses the 5V model to explain why infrastructure had to evolve:

VMeaningSimple example
VolumeLarge amount of dataBillions of log records or transactions
VelocityHigh speed of data generation or arrivalReal-time sensor streams or clickstreams
VarietyDifferent data formats and structuresTables, images, emails, JSON events

The slide then links this growth in data demands to the need for virtual machines and warehouse-scale computers, which are both ways to make computing more scalable and efficient.

Key Takeaway
The computing foundations section teaches the stack beneath applications: hardware -> operating system -> applications -> users, and it explains why data growth pressures this stack to scale.

7. Scaling, Virtualization, Containers, and Cloud

7.1 Virtual machines

A virtual machine (VM) lets a guest operating system use computing resources provided by a host operating system and a virtualization layer. The key idea is isolation plus better utilization of hardware.

Diagram walkthrough: VM stack

LayerWhat the slide showsWhy it matters
Host computer hardware The physical CPU, memory, keyboard, disk, and devices. This is the real machine underneath everything.
Host operating system The main OS installed on the physical machine. It manages the real hardware and supports virtualization.
Virtualization layer The layer that creates virtual CPUs, virtual memory, and virtual disks. It makes one physical machine appear as multiple isolated machines.
Guest operating systems Separate OS instances, each with its own applications and virtual resources. Different workloads can be isolated while sharing the same physical hardware.
Example: A company may run a Windows server VM for one legacy application and a Linux VM for another service on the same physical host.

7.2 Warehouse-scale computers

The deck uses the term warehouse-scale computers to describe very large computing environments built from many servers working together. The focus shifts from optimizing one machine to optimizing an entire fleet.

The slide highlights five design goals:

  • Performance per dollar of capital invested
  • Performance per joule of energy consumed
  • Reliability
  • Request-level parallelism
  • Handling device failures

Diagram walkthrough: single server -> rack -> cluster

Level shownWhat it representsMain lesson
One server A single machine with its own local memory and disk. Fast local access, but limited scale.
Local rack Many servers connected through a rack switch. More capacity through parallel resources, with some network overhead.
Cluster / datacenter Multiple racks connected through higher-level switching. Massive capacity, but distributed design must account for latency and failure.
Example: A search engine serving millions of queries cannot rely on one machine. It distributes requests across many servers and tolerates failures as a normal operating condition.

7.3 Containers

The deck contrasts containers with virtual machines:

  • VMs provide hardware virtualization.
  • Containers provide OS-level virtualization.

The practical consequence is that containers are usually lighter-weight when the goal is to isolate applications rather than run full guest operating systems. A container packages an application and its dependencies without bundling an entire extra operating system.

Why containers help

  • They reduce the “it works on my machine” problem.
  • They package runtime dependencies together.
  • They make deployment more consistent across environments.
Example: If an app needs a specific Python version and a specific library version, a container can package those together so the deployment server behaves like the developer’s machine.
The slide’s Docker analogy - “like a zip file for applications” - is not literally precise, but it is a good study shortcut: a container packages application code plus the environment it expects.

7.4 Cloud computing: core characteristics

The deck lists five defining characteristics of cloud computing:

CharacteristicWhat it means in practiceSimple example
On-demand self-service Users can provision resources when needed without waiting for manual setup. A developer launches a server from a web console.
Broad network access Services are reachable over the network using standard mechanisms. Accessing storage or compute through the internet.
Resource pooling Provider resources are shared among multiple users. Many customers using one provider’s large datacenter pool.
Rapid elasticity Resources can scale up or down quickly. Adding more servers during a traffic spike.
Measured service Usage is metered and tracked. Billing by CPU time, storage, or bandwidth used.

7.5 Cloud deployment and service models

The deck divides cloud models in two ways:

Model TypeOptions from the slideWhat they mean
Access / deployment models Private, Community, Public, Hybrid These describe who owns the infrastructure and who shares it.
Service models IaaS, PaaS, SaaS These describe how much of the stack the provider manages for you.

Quick distinctions

  • IaaS (Infrastructure as a Service): rent computing infrastructure such as virtual machines, storage, and networking.
  • PaaS (Platform as a Service): use a managed platform where the provider handles more of the runtime and deployment environment.
  • SaaS (Software as a Service): consume finished software over the network.
Example: Renting a VM is closer to IaaS; deploying code onto a managed app platform is closer to PaaS; using a browser-based office suite is closer to SaaS.

7.6 Cloud implementation and APIs

The final cloud slide says cloud systems hide underlying hardware details and expose services through APIs. This means developers interact with abstractions like compute, networking, and storage, rather than directly wiring physical machines.

Diagram walkthrough: abstraction in cloud platforms

LayerWhat the figure communicatesStudy meaning
Users / internet-facing applications Applications sit above the cloud platform and call into it. Developers consume services instead of managing raw hardware directly.
APIs Programmatic interface between applications and the platform. Cloud resources are requested and controlled through software calls.
Cloud platform services Compute, networking, and storage are presented as managed services. The platform abstracts and organizes the underlying infrastructure.
Underlying standard hardware Physical servers still exist beneath the platform. The cloud does not remove hardware; it hides and pools it.

The slide names OpenStack as an industry standardization effort, reinforcing the idea that cloud environments are built around consistent interfaces and resource abstractions.

Key Takeaway
To scale modern systems, organizations move from single machines to virtualized resources, containerized applications, and cloud-managed infrastructure that can be provisioned, isolated, and expanded more efficiently.

8.2 Fast answer templates

  • If asked how security failure happens: “A threat exploits a vulnerability because controls are missing, weak, or misconfigured.”
  • If asked why encryption is mixed in practice: “Asymmetric encryption solves secure key exchange; symmetric encryption handles bulk data efficiently.”
  • If asked why organizations use redundancy: “To preserve availability when components fail or traffic spikes.”
  • If asked why cloud matters: “It abstracts hardware, exposes resources via APIs, and supports elastic, pooled, measured access to compute resources.”

8.3 Final summary of the full deck

This deck teaches that secure digital systems require both security logic and infrastructure logic. First, you define the security goals (CIA), identify vulnerabilities and threats, and apply controls such as patching, firewalls, encryption, signatures, training, and redundancy. Then, you understand the computing platforms those controls live on: the OS, virtual machines, containers, warehouse-scale systems, and cloud platforms.

Key Takeaway
The most exam-ready one-line summary is: Protect information with the right controls, then run those controls on infrastructure that can scale, isolate, and stay available.
